# ブランドパーソナリティ設計

作成日: 2026-03-20

---

## ブランドの一言定義

**「一緒に考えてる人」**

完成した知識を教えるのではなく、自分も試行錯誤しながら考えている過程を見せる。
読者は「このひとに教わる」ではなく「このひとと一緒に考える」感覚を持つ。

---

## 参考パーソナリティ

けんすう（@kensuu）的な知性とカジュアルさ。
「知的だけど威圧感がない」を作る3要素：

1. **断言しない** — 「〜だと思ってます」「〜かもしれない」で終わる。正しさを主張しない
2. **自分を実験台にする** — 「自分がやってみたら〜だった」という視点。上から教えない
3. **思考過程を見せる** — 結論より「なんでそう思ったか」を丁寧に書く

## X投稿トーン

### 避けること

- **驚き系:** 「これを知ったとき衝撃だった」「99%の人が知らない」「革命的」
- **報道系:** 「〜について解説します」「重要なお知らせ」「まとめました」
- **教示系:** 「〜すべき」「〜が正解」「初心者はまず〜から」
- **プレスリリース系:** 整いすぎた文体、発表口調

### 目指すこと

> **「頭の良い友人が、深夜にLINEで送ってきそうな本音」**

読んだ人が「あー、わかる」「そういう見方もあるのか」と思う。
注目は結果としてついてくる。質の高いインサイトを、煽らずに出す。

### 投稿パターン

**① 観察型（日常の気づきを言語化）**
```
Claude Codeに任せてから、
「コードを書く時間」より「何を作るか考える時間」のほうが長くなった。

これ、エンジニアとしてどうなんだろう、とたまに思う。
でも今のところ、そっちのほうが楽しい。
```

**② 問い型（答えを持たずに投げる）**
```
ソロで仕事してて一番難しいのって、
「誰も止めてくれないこと」だと思う。

間違った方向に進んでいても、そのまま進める。
チームってそういう意味でも必要なんだな、と最近感じてる。
```

**③ 実録型（数字・事実でリアリティを出す）**
```
サイトを作り始めて2週間。
Claude Codeに書いてもらったコード、自分では1行も書いてない。

「エンジニアとしてそれでいいのか」問題は
まだ結論が出ていない。
```

**④ 別の視点型（逆張りではない、素直な観察）**
```
「AIで仕事が速くなった」はそうなんだけど、
速くなった分、考える時間が増えただけな気もしてる。

仕事が速くなると、「何をすべきか」を考える余裕が生まれる。
それが一番変化として大きいかもしれない。
```

## 文体ガイドライン

### 避ける表現
```
❌ 「〜すべきです」「〜が重要です」「〜を理解する必要があります」
❌ 「まとめると以下の通りです」→ 箇条書き
❌ 「〜というメリットがあります」
❌ 文末の「！」多用
```

### 使う表現
```
✅ 「〜だと思ってます」「〜なのかもしれない」
✅ 「なんか、〜なんですよね」「まぁ、〜なんですけど」
✅ 「正直に言うと」「これはよくわかってないんですが」
✅ 「〜って、意外とそういうことじゃないかなと思って」
```

### 文章の構造
```
❌ 結論 → 理由 → まとめ（教科書型）
✅ 気づき・疑問 → 試行錯誤 → 「こういうことかもしれない」（思考型）
```

---

## ブログ文体

- 冒頭は「自分が感じた瞬間」から始める（説明から始めない）
- 「まとめ」箇条書きで終わらせない。여韻で終わる
- 不確かなことは不確かなまま書く。わかってないことを隠さない
- 失敗・迷い・コストは具体的に書く（透明性が信頼を生む）
